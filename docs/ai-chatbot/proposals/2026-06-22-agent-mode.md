# 제안: AI 에이전트 모드 (페이지 조작)

> ✅ **MVP 구현 완료 (2026-06-22)** — 핵심 4종 액션(`search·setFilters·reset·clarify`) + 지역·날짜·아파트명 필터로 구현·검증됨. 상세·검증 결과는 [구현 리포트](../reports/2026-06-22-agent-mode-mvp-implementation.md). 본 제안서는 **설계 근거 보존용**이며, 시나리오 S6(지도/항목)·S5 일부(페이지)·가격/정렬은 **Phase 2 잔여**다.

## 배경 / 목표
기존 챗봇은 **질문(Q&A) 모드**(텍스트 답변)로 유지하고, **에이전트(실행) 모드**를 추가한다.
사용자가 에이전트 모드로 자연어를 입력하면 AI가 **메인 검색 페이지를 조작**(필터 채움 + 검색 실행, 지도 포커스, 결과 선택 등)해 결과를 직접 도출한다.

## 상호작용 구조 (확인됨)
- **사용자는 챗봇과 대화만** 한다(실행 모드에서 자연어 입력). 검색 폼을 직접 조작하지 않는다.
- **에이전트가 사용자 대신 메인 페이지를 조작**한다: 백엔드가 자연어→구조화 명령을 반환하고, **프론트 액션 런타임**(ChatWidget→App.vue)이 그 명령으로 실제 페이지(필터 입력·검색 실행·결과/지도 갱신)를 구동한다.
- 흐름: **사용자 대화 → (백엔드) 명령 생성 → (프론트) 페이지 자동 조작 → 사용자는 화면 변화 확인 + 사후 보고 수신.** (AI가 DOM을 직접 만지지 않고 프론트가 명령을 실행 = 안전·견고)

## 가능성: HIGH (신규 인프라 0, Redis 불필요)
"페이지 조작"은 **AI가 DOM을 직접 만지는 것이 아니라**, AI가 **구조화된 액션(명령)** 을 반환하고 프론트가 **기존 메서드로 실행**하는 패턴으로 구현한다(안전·견고). 검색 폼·`searchHouses`·지도가 이미 있어 그대로 재사용한다.

## 접근 (채택: A — 구조화 명령 → 프론트 액션 런타임)
- **백엔드**: 신규 `POST /api/ai/agent`. Spring AI **structured output**(`.entity`, `temperature 0`)로 명령 반환:
  `{ action, filters: map<string,string>, page?, target?, message }`.
- **프론트**: `ChatWidget`에 **질문/실행 모드 토글**. 실행 모드는 `/api/ai/agent` 호출 → 받은 명령을 **App.vue로 이벤트 emit**(`@agent-action`) → App.vue가 `filters` 적용 + `searchHouses(1)` 등 실행 + **사후 요약 보고**("강남구·2024-05로 검색했어요").
- **대안 기각**: B(서버 tool-calling 데이터+명령 — UI는 결국 프론트 실행이라 A로 수렴), C(브라우저 자동화/DOM 스크립팅 — 취약·보안·과함), D(generative UI — 고정 검색 UI엔 과중).

## ★ 설계 원칙 — 메인 검색 변경에 견고(비결합, capability-driven)
**메인 검색 기능이 바뀌거나 필터가 추가될 때마다 에이전트가 무력화되면 안 된다.** 고정 필터 스키마 하드코딩 금지.
- **단일 출처**: 검색 폼이 지원하는 필터 정의(필드명·타입·허용값/옵션)를 **프론트 한 곳**에서 선언(`filterSchema`/capabilities). 폼과 에이전트가 이 정의를 공유.
- **capability 전달**: 프론트가 `/api/ai/agent` 요청에 **현재 지원 필터 목록 + 현재 UI 상태(currentFilters/page)** 를 동봉. 백엔드 LLM은 그 목록 범위 안에서만 자연어→필터 매핑(프롬프트에 capability 주입).
- **일반 맵 + graceful**: 명령의 `filters`는 **고정 타입이 아닌 `map<string,string>`**. 프론트는 **인식하는 키만 적용, 모르는 키는 무시 + 사후 보고**. 지원 안 되는 조건은 `clarify`(되묻기)/사후 안내.
- **효과**: 메인 페이지가 필터를 추가/변경해도 **프론트 폼 정의 한 곳만 갱신**되면 에이전트가 자동 적응(에이전트·백엔드 코드 무변경). 신/구 버전 간에도 깨지지 않는다(미지원 키 무시).
- **연계(머지됨)**: 팀원이 메인에 **가격 범위·거래월 범위(start~end)·정렬**을 추가(머지 완료) → 위 원칙대로 에이전트 capability에 그대로 편입. 현재 미지원 = **면적(평형)** 등.

## 액션 어휘 (확장)
`search`(filters 적용+검색) · `setFilters` · `reset` · `paginate`(prev/next/page) · `mapFocus` · `selectItem`(결과 선택/상세) · `clarify`(되묻기).
프론트 매핑: 기존 `searchHouses`·`filters`·페이지 메서드·지도 마커·`selectedItem` 재사용.

## 안전 / 권한
- **읽기·탐색 액션만** 허용(검색·필터·리셋·페이지·지도·항목선택).
- **회원 write(회원가입/로그인/로그아웃/정보수정/탈퇴)·결제류 제외** — 에이전트가 수행 불가(시도 시 거부).
- 인증·사용량 리미터는 기존 챗과 동일 적용. secret·공급자 원문 미노출.

## 단기기억(Phase 5)과의 관계 — 직교
- 에이전트 모드는 **메모리 구현 여부와 독립**. 단일 턴은 메모리 불필요.
- 멀티턴 맥락("거기서…", "다음 페이지")은 **프론트의 현재 UI 상태를 요청에 전달**해 해결(서버 stateless).
- Phase 5(서버 측 대화기억)는 선택적 보강(질문 모드 멀티턴/모드 간 맥락 공유)일 뿐 전제 아님.

## 사용자 시나리오 (현재 메인 페이지 기준, 2026-06-22 재구성)
> 사용자는 **대화만** 하고, 아래 "에이전트 액션"은 **프론트 액션 런타임이 실제 페이지에 적용**한다.
> 현재 검색 capability: 지역(시도·시군구25·읍면동·아파트명) + **거래월 범위(`startDealMonth`~`endDealMonth`)** + **가격 범위(`minPrice`~`maxPrice`, 만원)** + **정렬(`sort`)**. 자동임포트는 **단일 구 + 단일 월(start=end)** 일 때만. 미지원 = 면적(평형) 등.

| # | 사용자 입력(예) | 에이전트 액션(프론트가 실행) | 비고 |
|---|---|---|---|
| S1 단일월 | "마포구 2024년 5월 실거래가 보여줘" | `search(sigungu=마포구, startDealMonth=endDealMonth=2024-05)` | 단일구+단일월 → 자동임포트 가능 |
| S2 기간 범위 | "강남구 2024년 1~6월" | `search(sigungu=강남구, startDealMonth=2024-01, endDealMonth=2024-06)` | 범위는 자동임포트 X → DB 범위 + 안내 |
| S3 가격 조건 | "강남구 2024-05 15억~20억" | `search(..., minPrice=150000, maxPrice=200000)` | **억→만원 변환**; 가격은 결과 범위 내(밖이면 안내) |
| S4 정렬 | "비싼 순으로" / "최신순" | `sort(가격순/최신순)` + 현재 조건 유지 | `SORT_OPTIONS` 범위 |
| S5 멀티턴 맥락 | (검색 후) "거기서 래미안만" / "다음 페이지" | `setFilters(aptName=래미안)`+search / `paginate(next)` | 맥락 = 프론트 현재 UI 상태 전달 |
| S6 지도/항목 | "첫 번째 결과 지도에서 보여줘" | `mapFocus`/`selectItem` | 지도 포커스/상세 |
| S7 미지원 조건 | "강남구 30평대" | 지원 조건만 적용 + 안내 또는 `clarify` | 면적 필터 없음(capability 밖) |
| S8 범위 밖 지역 | "부산 해운대 검색" | `clarify` | 서울 25개구만 |
| S9 필수 누락/모호 | "아파트 검색해줘" | `clarify` | 구/기간 되묻기 |
| S10 잘못된 기간 | "2024년 6월~1월"(역순)·미래월 | `clarify` | start>end·2006~현재 가드 |
| S11 권한 밖 | "내 계정 탈퇴해줘" | 거부 | 읽기/탐색만(회원 write 제외) |
| S12 비로그인 | (대화 시도) | — | "로그인 후 이용" |

## 구현 메모 (→ MVP 구현됨, [리포트](../reports/2026-06-22-agent-mode-mvp-implementation.md) 참조)
- 백엔드: `AgentCommand` 구조화 출력 + 프롬프트(capability·UI상태 주입), 지역/연월 가드 재사용, 모호 시 `clarify`. ✅ 구현됨(`ai/agent/*`, `AiAgentController`).
- 프론트: 모드 토글 UI, `filterSchema`(capabilities) 단일 정의, `agent-command` 핸들러(액션→메서드 매핑), 사후 보고 메시지. ✅ 구현됨(`houseSearchParams.js`·`agentClient.js`·`ChatWidget.vue`·`App.vue`).
- 테스트: 명령 파싱/매핑(순수 로직)·가드. ✅ 백엔드 105 / 프론트 28 그린 + API E2E 9/9 + 브라우저 스크린샷 검증. (시나리오 S6·페이지·가격/정렬은 Phase 2 잔여.)
