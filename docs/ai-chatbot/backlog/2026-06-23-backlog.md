## 남은 작업

### AI 어시스턴트 tool calling 혼합형 재설계 (✅ 구현 완료)
[제안서/타당성/계획](../proposals/2026-06-23-tool-calling-assistant.md) · [트러블슈팅](../troubleshooting/2026-06-23-agent-intent-mapping-and-capability-drift.md).
- **동기**: 의도 매핑 오분류(불만→paginate, 검색→setFilters, 거래월 환각)가 `.entity` 강제 생성 구조 + 프롬프트 땜질의 한계.
  + 전월세 capability drift. + 모드 토글 UX.
- **방향**: 단일 `/api/ai/assistant`에서 LLM tool calling — 질문=텍스트 답변(데이터 tool), 조작=프론트 액션 tool(`returnDirect`).
  기존 `AgentCommand`/가드/프론트 핸들러 재사용, 모드 토글 제거. 채택 경로 (A) `@Tool(returnDirect=true)`.
- ✅ **Phase 0 PoC**(게이트): returnDirect 실측 통과 — 액션 tool 반환 `AgentCommand`가 content(JSON)로 오고 `finishReason="returnDirect"`로 결정적 식별. B안 불필요.
- ✅ **Phase 1** 백엔드 통합 `/assistant`(`AiAssistantController`) + `PageActionTools`(6 액션 tool, 가드 내장) + `AiProviderErrors`/`AiRequests` 공통화 + `HouseTools` 전월세(dealMode) 확장.
- ✅ **Phase 2** 프론트 통합(`parseAssistantResponse` type 분기) + filterSchema 전월세 키 + **폼↔schema 동기화 테스트**(drift 재발 차단).
- ✅ **Phase 3** 모드 토글·`/agent`·`/chat`·레거시 파서 제거, `/assistant` 기본 활성. 트랜잭션 브랜치에서 전체 검증 후 master 1회 머지.
- **검증**: backend 144 / frontend 44 그린 + docker·브라우저 실측(질문/검색/전세/월세/페이지/불만/비서울/멀티턴). 단기기억은 같은 ChatClient 빈으로 자동 승계.
- ✅ **후속 ①(완료)**: 검색 폼 기본 거래월을 최신월(직전월)로 세팅(`currentDealMonth`) → 거래월 미지정 검색(예: AI '강남구 검색')이 라이브 조회 없이 0건 되는 것 방지.

#### 재설계 후속 개선 (별도 — returnDirect 단일 액션 모델의 한계)
직접 사용 중 드러난 "맥락 이해 ↔ 동작" 괴리. 단일 동작은 개선됐으나 결과 기반·순차 동작은 미해결.
- ⬜ **② 결과 기반 선택** — "면적 가장 작은 매물 선택" 등은 현재 결과 목록(아파트명·면적·가격)이 모델에 전달되지 않아 환각(1번 선택). assistant 요청에 현재 결과 요약(순번·아파트명·면적·가격)을 동봉하고 프롬프트에 노출 + 면적 정렬 옵션(sort) 추가 필요.
- ⬜ **③ 다중 동작** — "3페이지 이동 후 2번째 선택" 같은 순차 동작은 `@Tool(returnDirect=true)`가 1액션/호출이라 첫 동작만 실행. 해결: B안(`internalToolExecutionEnabled(false)`+`ToolCallingManager` 수동 루프로 멀티 tool 수집) 또는 `AssistantResponse.commands[]` 배열 반환. 구조 변경 규모 큼.

### Phase 5 대화기억 (✅ 머지 완료 — 세션 휘발 방식)
backend [#28](https://github.com/repechage-team/no-home-backend/pull/28) / frontend [#12](https://github.com/repechage-team/no-home-frontend/pull/12).
- **구현**: 중앙 `ChatClient` 빈에 `MessageWindowChatMemory`(**InMemory**, window 10) + `MessageChatMemoryAdvisor` 부착 → `/chat`·`/agent` 자동 적용. 대화 키 = `memberId:<프론트 세션 UUID>`(memberId 격리 + 세션 UUID 분리, 누락 시 `default`). 프론트는 `getConversationId()`(sessionStorage UUID, 질문·에이전트 공유).
- **세션 휘발(영속 안 함)**: 초기에는 JDBC 영속(`conversationId=memberId`)을 검토했으나, 질문/답변 원문 영속 저장이 로깅·개인정보 정책과 충돌하고 "닫으면 초기화"가 운영상 더 안정적이라 **InMemory + sessionStorage UUID**로 결정. 탭/세션을 닫으면 새 id가 되어 대화가 초기화된다.
- **검증**: backend 126 / frontend 50 그린. 브라우저 실측 — 같은 세션 멀티턴 유지("마포구"), 다른 세션 격리("물어본 적 없음"), 세션 종료 모사 시 새 UUID 발급.
- **재설계와의 결합**: 같은 ChatClient 빈을 쓰므로 통합 `/assistant`에도 자동 승계 → 멀티턴 자연스러움↑.
