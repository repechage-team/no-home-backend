# 문서 규약/위치 전략 — 결정 대기

> 이 문서는 결정문이 아니라 **현재 상황과 선택지를 사실대로 기록**한 노트다. 표준 문서 규약·위치는 **새 팀이 결정**한다.

## 상황
- **`no-home-artifact` 레포**: 통합 실행(`docker-compose.yml`) + 프로젝트 문서 허브로 구성됨. 기존 문서(`docs/PRD.md`·`spec.md`·`plan.md`·`sprints/Sprint0~12.md`·`working-memory.md` + `AGENTS.md`의 Manager/Generator/Reviewer 운영 모델)는 **이전 프로젝트 오너의 자료이며, 새 팀을 위한 온보딩 용도로 공유된 것** — **새 팀이 합의한 규약은 아님**.
- **팀 재편**: 대규모 인원 변동으로 신규 2인 합류(작성자 포함). go-forward 문서 표준은 미정.
- **현재 챗봇 문서**: `no-home-backend/docs/`에 별개 구조(도메인 디렉터리 + `_shared` + `guide`/`templates` + frontmatter + 날짜 프리픽스)로 존재. 이 또한 **새 팀 합의 규약은 아님**. 현재 backend 기능 브랜치에 로컬 커밋됨(아직 미머지).
- 즉 **두 문서 체계가 공존하며 둘 다 미합의** 상태.

## 결정 대기 (새 팀)
1. **표준 문서 규약**: ① artifact의 Sprint/working-memory/AGENTS 모델 유지·계승, ② backend 챗봇 문서의 도메인/`_shared`/frontmatter 구조, ③ 새 하이브리드.
2. **문서 위치**: ① artifact 단일 허브, ② 코드 레포별 분산, ③ 허브(교차)+레포별(구현 노트) 하이브리드.
3. **챗봇 문서 정리 방향**: 채택 규약으로 재작성/이관할지, 현재 형태 유지할지.

## 트레이드오프 (요약)
- **artifact 허브**: 통합 레포 취지에 부합, 단일 조망. 비용 = 규약 재합의 필요(기존이 미합의 온보딩 자료), 코드↔문서 원거리, 다중 레포 링크는 URL.
- **backend 유지(현행)**: 마찰 0·코드 근접. 단 프론트/교차 문서가 백엔드 레포에 종속, 통합 레포 취지와 불일치.
- **하이브리드**: 소유권 명확. 단 교차/레포-로컬 판단 비용, 링크 정책 필요.
- **타이밍 이점**: 챗봇 문서는 아직 **기능 브랜치에만**(미머지) 있어, 위치를 **머지 전에 정하면 재정리 비용이 낮음**(참고: [git 이력 비밀정보 잔존 이슈](issues/2026-06-21-git-history-secret-exposure.md) — 한 번 기본 브랜치에 들어가면 이력 정리가 어렵다는 교훈).

## 잠정 권고 (비구속)
- 새 팀이 문서 규약을 합의한 뒤, **합의 규약으로 챗봇 문서를 artifact 허브로 정리**하는 방향을 권한다(통합 레포 취지·단일 조망).
- **합의 전에는 `no-home-artifact`를 건드리지 않는다**(타인 온보딩 자료·미합의). 그때까지 챗봇 문서는 현재 위치(backend)에 둔다.

## 다음 행동
- 새 팀 문서 규약 논의 안건으로 상정. 결정 시 본 문서를 `accepted`로 갱신하고 이관/재작성 작업을 별도 진행.
