# ISSUE: Git 이력의 비밀정보 및 생성 파일 잔존

## 증상
- 백엔드와 프론트엔드 저장소에서 로컬 비밀 설정 파일인 `.env`가 Git 추적 대상이었다.
- 백엔드 `target/`, 프론트엔드 `node_modules/`·`dist/`도 함께 추적되어 소스 외 생성 파일이 저장소 대부분을 차지했다.
- 최신 작업 트리에서 추적을 해제해도 과거 커밋에는 `.env`와 생성 파일의 blob이 남아 있어 커밋 이력을 통해 다시 조회할 수 있다.

실제 키 값은 이 문서에 기록하지 않는다. 확인된 범위는 백엔드의 DB/API/지도 관련 민감 환경변수와 프론트엔드의 지도 API 환경변수이며, 새 GMS 키는 커밋 이력에 포함되지 않았다.

## 재현 방법
1. 정리 전 커밋에서 `git ls-files .env target node_modules dist`를 실행한다.
2. 백엔드는 `.env` 1개와 `target/` 103개, 프론트엔드는 `.env` 1개와 `node_modules/` 659개 및 `dist/` 4개가 추적된 것을 확인한다.
3. 최신 커밋에서 추적이 해제된 뒤에도 `git log --all -- .env` 또는 과거 커밋 조회로 `.env` 이력이 남아 있음을 확인한다.

## 근본 원인
- 초기 저장소 구성 시 `.gitignore`가 없거나 이미 추적된 파일에 ignore 규칙만 추가해 인덱스에서 제거되지 않았다.
- `.env.example`과 실제 `.env`의 역할이 분리되지 않은 상태로 초기 소스가 커밋됐다.
- 프론트엔드는 `package-lock.json`으로 의존성을 재현할 수 있는데도 설치 결과인 `node_modules/`가 커밋됐다.
- 백엔드와 프론트엔드 빌드가 각각 `target/`, `dist/`를 재생성할 수 있는데도 산출물이 커밋됐다.

## 영향 범위
### 영향 받는 것
- 저장소 접근 권한이 있는 사용자가 과거 커밋에서 기존 환경변수 값을 확인할 수 있다.
- 프론트엔드 저장소의 Git pack에 의존성 파일이 남아 clone·fetch 크기가 불필요하게 증가한다.
- 단순 추적 해제 커밋만으로는 과거 이력과 기존 clone·fork·캐시에서 값을 제거할 수 없다.

### 영향 받지 않는 것
- 로컬 `.env`, `target/`, `node_modules/`, `dist/`는 추적만 해제했으므로 현재 개발 환경에서는 그대로 사용할 수 있다.
- 새 GMS 키는 기존 커밋에 포함되지 않았고 `.gitignore`로 차단됐다.

## 현재 조치
- 백엔드 정리 커밋 `07f45f4`: `.env`, `target/` 추적 해제 및 `.gitignore` 보완.
- 프론트엔드 정리 커밋 `ff334ea`: `.env`, `node_modules/`, `dist/` 추적 해제 및 `.gitignore` 생성.
- 두 저장소 모두 최신 인덱스 기준 금지 경로 추적 0건, 예상 밖 삭제 0건을 확인했다.
- 정리 커밋은 현재 로컬 브랜치에만 있으며 아직 원격에 푸시하지 않았다.

## 결정된 진행 방향: 키 교체 없이 추가 노출 차단
현재 단계에서는 키를 교체하지 않고 Git 이력에서 비밀 파일을 제거해 접근 가능성을 줄이는 방향을 검토한다. 이 방식은 이미 생성된 clone·fork·캐시에서 값을 회수하지 못하므로 **완전한 비노출이나 키 폐기를 보장하지 않는다.**

1. 원격 저장소 작업을 일시 동결하고 진행 중인 브랜치·태그·PR·팀원 로컬 작업을 확인한다.
2. 백엔드 전체 이력에서 `.env`, `target/`을 제거한다.
3. 프론트엔드 전체 이력에서 `.env`, `node_modules/`, `dist/`를 제거한다.
4. 유출 커밋이 공통 이력에 있으므로 기능 브랜치만이 아니라 관련 기본 브랜치와 태그도 함께 재작성한다.
5. 재작성된 참조를 원격에 강제 갱신하고 팀원은 새로 clone한다. 기존 clone은 안전하게 폐기한다.
6. 지도/API 키에는 가능한 허용 도메인·허용 API·쿼터 제한을 적용한다.

## 완료 기준
- 새 clone에서 `git ls-files .env target node_modules dist` 결과가 0건이다.
- 모든 관련 참조에서 `git log --all -- .env` 결과가 0건이다.
- Git 객체 경로 검사에서 `.env`, `target/`, `node_modules/`, `dist/`가 발견되지 않는다.
- 비밀정보 스캔에서 실제 키 값이 검출되지 않는다.
- 팀원이 재작성된 이력을 기준으로 재클론했고 기존 작업 이전을 완료했다.

## 갱신 (2026-06-22): 키 교체 + 구 키 revoke 완료 → 노출 무력화
- 당초 "키 교체 없이 이력만 정리" 방향이었으나, **키를 실제 교체(rotation)** 하고 **구 키를 제공자 포털에서 폐기(revoke)** 했다.
- 효과: 이력·기존 clone·fork·캐시에 구 키 문자열이 남아 있어도 **해당 키는 더 이상 인증되지 않아 무력화**된다. 즉 **라이브 비밀 노출 위험은 해소**.
- 남은 작업의 성격 전환: 이력 재작성은 이제 "비밀 노출 대응"이 아니라 **저장소 위생/용량 정리**(`.env`/`target`/`node_modules`/`dist` 제거)다. 파괴적·팀 조율 필요 작업이므로 별도 일정으로 진행.
- `status`: history-rewrite 미완이라 `in-progress` 유지(단, 노출 위험은 해소됨).

## 상태/메모
- **완료:** 최신 인덱스 추적 해제, `.gitignore` 정비, 저장소별 정리 커밋 분리, **키 교체 + 구 키 revoke (2026-06-22)**.
- **대기:** (위생) 전체 이력 재작성 여부 확정, 협업 영향 확인, 원격 강제 갱신.
- **참고:** 노출 위험 자체는 revoke로 해소됨. 이력 재작성은 위생/용량 목적.
